git工作区、暂存区、版本库

工作区就是你在电脑里能看到的目录。

暂存区英文叫stage, 或index。一般存放在 “.git目录下” 下的index文件（.git/index）中，所以我们把暂存区有时也叫作索引（index）。

版本库:工作区有一个隐藏目录.git，这个不算工作区，而是Git的版本库。

git基本操作

git中文件的三种状态

  • 已提交(committed):已提交表示数据已经安全的提交到本地数据库中。
  • 已修改(modified):已经修改了文件，但是还没有保存到数据库中。
  • 已暂存(staged):对一个已经修改的文件做了当前版本的标记，使其包含在下次提交的快照中。

git项目的三个工作区域

  • git仓库：保存项目的元数据与对象数据库的地方。当从计算机拷贝仓库时，拷贝的数据就是这些。
  • 工作目录：是对项目的某个版本独立提取出来的内容，这些从git仓库的压缩的文件中提取出来的文件，放在磁盘上供使用者使用与修改。
  • 暂存区域：暂存区域作为一个文件，保存下次将要提交的文件列表信息。一般放在git的仓库目录中。

git工作的基本流程

  • 在工作目录中修改文件。
  • 将文件的快照放入暂存区域保存。
  • 提交更新，也就是提交暂存区域的文件的新的内容。

版本控制

版本控制(VCS)是一种记录一个或者若干个文件内容变化，以便将来查询特定版本修订情况的系统。

本地版本控制

本地版本控制大多采用某种简单的数据库来记录文件的历次更新差异。如下图展示的一种版本控制方法：

集中化的版本控制系统

集中化的版本控制系统(CVCS)是为了解决不同系统的开发者协同工作的，集中化的版本控制系统存在一个单一的集中管理的服务器，保存所有文件的修订本，而协同工作的人们通过客户端连接到这台服务器，取出最新的文件或者提交更新。集中化的版本控制如下图：

集中化的版本控制系统可以方便管理各个开发者的权限，并且管理一个CVCS要比远在各个客户端上维护本地数据来得轻松些。CVCS 的缺点也是明显的：中央服务器的单点故障。

分布式版本控制系统

分布式版本控制系统(DVCS)中，客户端并不只提取最新版本的文件快照，而是把代码仓库完整地镜像下来。即使协同工作的服务器宕机，事后可以通过任何一个镜像来恢复，每一次的克隆操作，实际上是一次对代码仓库的完整备份。分布式版本控制系统如下图：

Git基础

直接记录快照，而非差异比较

Git和其他版本控制系统的主要差别在于git对待数据的方法，概念上讲，其他大部分系统以文件变更列表的方式存储信息。这类系统讲他们保存的信息看作是一组基本文件和每个文件随时间逐步累积的差异。如下图展示的文件差异存储方式：

git不按照上述的方式来保存数据，git更像是把数据看作是对小型文件系统的一组快照。每次提交更新，或者保存项目状态时，它主要对当时的全部文件制作一个快照并保存这个快照的索引。为了高效，如果文件没有更改，git不再重新存储文件，而是保留一个连接指向原来的文件，git对待数据更像是一个快照流。快照流方式存储如下图：

近乎所有的操作都是本地执行

在git中的绝大多数的操作都只需要访问本地文件和资源，一般不需要来自网络上的其他的计算机的信息。在本地查询项目历史，在本地进行差异计算。按时提交，延时上传，方便的处理文件与项目。

git保证完整性

git中所有数据在存储前都计算校验和，然后以校验和来引用。git用以计算校验和的机制叫做**SHA-1散列**，校验和由40个十六进制字符组成的字符串。执行git操作，几乎只往git数据库中添加数据，很难让git执行任何不可逆操作，或者以任何方式清除数据。

三种状态

git有三种状态，分别是:

  • 已提交(committed) ：表示数据已经安全的保存在本地数据库中。
  • 已修改(modified)：表示修改了文件，但是没有保存到本地数据库中。
  • 已暂存(staged)：表示对一个已修改文件的当前版本做了标记，使之包含在下次提交的快照中。

三种状态分别对应三个工作区域的概念：

  • git仓库：用来保存项目的元数据和对象数据的地方。这是git最重要的部分，从其他计算机克隆仓库是，拷贝的就是这里的数据。
  • 工作目录：是对项目的某个版本独立提取出来的内容。这些从git仓库的压缩数据库中提取出来的文件，放在磁盘供用户使用和修改。
  • 暂存区域：是一个文件，保存了下次将要提交的文件的列表信息，一般在git仓库目录下中，有时候也被称作“索引”。

关于三个工作区域的关系展示如图：

基本的git工作流程：

  1. 在工作目录中修改文件
  2. 暂存文件，将文件的快照放入暂存区域
  3. 提交更新，找到暂存区域的文件，将快照永久性存储到git仓库目录。

初次运行git前的配置

git自带一个git config的工具帮助设置控制git外观和行为的配置变量。/etc/gitconfig文件：包含系统上每一个用户及他们仓库的通用配置。如果使用带有--system选项的命令时，它会从此文件读写配置变量。~/.gitconfig或者~/.config/git/config文件：只针对当前用户。可以传递--global选项让git读写此文件。当前使用仓库的git目录中的config文件(.git/config)：针对该仓库。在windows系统中，git会查找$HOME目录下的.gitconfig文件。

文件.gitignore的格式规范如下：

查看已暂存和未暂存的修改

想知道具体修改了什么地方，可以通过git diff来查看，结合git status可以更加方便查看相应改变。如果要查看已暂存的将要添加到下次提交里的内容，可以用git diff --cached来命令查看，甚至可以用git diff staged来查看。

提交更新

将暂存区域的文件提交，首先要确保还有什么修改过的或者新建的文件没有通过git add加入到暂存区，否则提交后看不到相应的变化，这些修改只保留在本地磁盘，需要通过git add添加到暂存区。最稳妥的方式每次使用git status来查看状态。通过使用git commit -a ....使得直接跳过git add的步骤，并且同样达到相应的效果。
